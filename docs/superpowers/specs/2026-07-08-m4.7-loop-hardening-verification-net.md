## §3 Mock-Harness — Entscheidungen

- **D1 — Injektion:** `test/mocks/homey.js` exportiert den SDK-Stub UND `installHomeyMock()`,
  das `Module._resolveFilename` patcht (`'homey'` → Mock-Pfad). Kein neues Package, kein
  Loader-Flag; Pattern ist auf Node 18–25 stabil.
- **D2 — HTTP stubben:** Tests laden zuerst `lib/VioletClient.js` und ersetzen
  `VioletClient.fetchReadings` durch einen Fixture-Resolver, DANN erst
  `drivers/pool/device.js` (dessen Destructuring greift die gepatchte Funktion). Kein Netz
  im Test.
- **D3 — Stub-Umfang (aufzeichnende Fakes):** `Device` mit `getSetting/setSettings`,
  `getStoreValue`, `hasCapability/addCapability/removeCapability/getCapabilities`,
  `setCapabilityValue` (Log: `{cap, value}[]`), `setCapabilityOptions` (Log),
  `registerCapabilityListener` (Map), `getAvailable/setAvailable/setUnavailable`,
  `log/error`, `this.homey` mit `flow.getDeviceTriggerCard` (Fake-Card: `registerRunListener`,
  `trigger`-Log), `setInterval/clearInterval` (**kein echtes Scheduling** — Tests treiben
  `_tick()` manuell), `manifest` (leeres `capabilities`-Objekt reicht für `_diagBaseTitle`).
- **D4 — Erste Tests** (`test/drivers/pool.device.test.js`), je gegen die 4 Fixtures:
  (K1) `onInit`+`_tick` wirft nicht, Device bleibt available; (K2) Capability-Bestand nach
  Tick == `desiredM2Capabilities`+Kern für die Fixture (Spot-Checks je Fixture, z. B.
  chlorine-only → `measure_chlorine` ja / `pump_speed_stage` nur bei Pump-Detektion);
  (K3) zweiter `_tick` mit gleicher Fixture = churn-frei (keine add/remove-Aufrufe);
  (K4) Apply-Regel: `setCapabilityValue` nur für vorhandene Caps, `undefined` übersprungen;
  (K5) 3× fetch-Fehler → `setUnavailable`.
- **D5 — Hook-Mapping:** `drivers/<id>/<file>.js` → `test/drivers/<id>.<file>.test.js`
  (flach unter `test/drivers/`, ein Punkt-Separator — konsistent mit dem flachen
  `lib/Foo.js → test/Foo.test.js`). Neuer Smoke-Test `test/hooks/run-matching-test.test.js`
  deckt beide Mappings + Fail-open ab (der Hook hatte bisher KEINE Tests).

## §4 Live-Smoke — Entscheidungen

- **D6 — Ort:** `scripts/live-smoke.js` (NICHT `test/` — darf nie in `npm test`/CI laufen:
  braucht echtes Gerät + Homey-CLI-Login; Node-Test-Runner-Discovery-Muster sicher umgangen).
- **D7 — Checks** (je `PASS`/`FAIL <Grund>`, Exit 1 bei ≥1 FAIL, read-only):
  - C1 Pool-Device gefunden (`homey api devices get-devices --json`, Filter App
    `de.neunbft.violet`, Driver `pool`) und `available`.
  - C2 Kern-Caps vorhanden (`measure_ph`, `measure_orp`, `measure_temperature`) UND
    Detektions-Konsistenz: für Gruppen mit erkannter Hardware aus dem frischen
    `getReadings?ALL` (via `detectFeatures`) existieren die Gruppen-Caps auf dem Device
    (Auto-Modus vorausgesetzt; Overrides → nur WARN, kein FAIL).
  - C3 Plausibilität frischer Werte: pH 4–10, ORP −200…1200 mV, Wassertemp 0–45 °C
    (nur wenn Wert nicht null).
  - C4 Frische: `CURRENT_TIME_UNIX` des Geräts vs. lokale Zeit < 15 min Drift; wenn Pumpe
    läuft und Warmup vorbei, ist `measurements_fresh` am Device true.
  - C5 Keine hängenden Alarme: für jedes `alarm_*` am Device gilt Device-Wert ⇒
    neu berechneter `buildM2Updates`-Wert aus dem frischen Raw (Device true, Neuberechnung
    false = hängend = FAIL; `alarm_water_balance` ausgenommen — LSI-abhängig).
- **D8 — Verankerung:** Punkt 8 in `release-readiness.md`: `node scripts/live-smoke.js`
  ausführen; Gerät nicht erreichbar = FAIL mit Klartext (Release-Prüfung braucht das

## §5 Stop-Hook — Entscheidungen

- **D9 — Ablauf** (`.claude/hooks/stop-verify.js`, Event `Stop`):
  1. stdin-JSON; `stop_hook_active === true` → exit 0 (Endlosschleifen-Schutz).
  2. `git status --porcelain` (spawn, fail-open bei Fehler); Zeilen filtern auf Pfade, die
     mit `lib/`, `drivers/` oder `.homeycompose/` beginnen (Rename-Syntax `a -> b`: Zielpfad
     zählt). Keine Treffer → exit 0 (sauberer Worktree = sofort durchwinken).
  3. Suite: `package.json scripts.test` via Shell (Muster/Env-Marker-Strip wie `test-gate`);
     kein `scripts.test` → überspringen.
  4. Validate: nur wenn `.homeycompose/app.json` existiert → `npx homey app validate`
     (Default-Level publish, B4); Spawn-Fehler = fail-open, roter validate = Befund.
  5. Befunde → exit 2 mit gesammeltem Output (Modell fixt weiter statt rot zu enden).
- **D10 — Tests** (`test/hooks/stop-verify.test.js`): Temp-Repos mit `git init` + Commit;
  Fälle: stop_hook_active → 0 · sauber → 0 · dirty `lib/` + rote Mini-Suite → 2 (Output
  enthält Suite-Fehler) · dirty `lib/` + grüne Suite → 0 · dirty NUR außerhalb (z. B.
  `docs/`) + rote Suite → 0 (gar nicht erst getestet) · kaputtes stdin → 0. Fixtures OHNE
  `.homeycompose` → der npx-validate-Zweig läuft in Tests nie (kein CLI-Download, CI-fest).
- **D11 — Registrierung ZULETZT** (B3): `"Stop"`-Block in `.claude/settings.json` erst im
  Abschluss-Commit, nachdem Suite+validate grün sind; in-Session-Verify per Direktaufruf
  (roter Fall über Temp-Fixture, grüner Fall gegen dieses Repo).
