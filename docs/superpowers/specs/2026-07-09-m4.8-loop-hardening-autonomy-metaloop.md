## §3 Hook-Telemetrie — Entscheidungen

- **D1 — Helper** `.claude/hooks/lib/log.js`: exportiert `logHook(hook, decision, cwd)`;
  hängt `{"ts":"<ISO>","hook":"<name>","decision":"block"|"pass"}` als eine JSONL-Zeile an
  `<cwd>/.claude/hooks/hook-log.jsonl` an; komplett in try/catch (fail-silent — Telemetrie darf
  NIE einen Hook brechen).
- **D2 — Pfadauflösung über `cwd`** (das Hook-Input-cwd), NICHT über `__dirname`: Hook-Smoke-Tests
  spawnen die Hooks mit Fixture-cwds ohne `.claude/hooks/`-Ordner → `appendFileSync` wirft →
  silent skip. So verschmutzen weder Fixture-Läufe das echte Log noch landet ein Log in Fixtures.
- **D3 — Instrumentierungsregel** („je ein Einzeiler" pro Entscheidungspunkt): `block` vor jedem
  `exit 2`; `pass` NUR auf dem „geprüft und für gut befunden"-Pfad (das finale `exit 0` nach
  bestandener Prüfung). Fail-open-/Nicht-zuständig-Exits loggen NICHT (sonst Rauschen von jedem
  Nicht-Commit-Kommando). PostToolUse-Sonderfälle: `run-matching-test` loggt `pass`, wenn es den
  gemappten Test tatsächlich ausgeführt hat; `json-guard` loggt `block` bei Befund (exit 2 ist
  dort Meldung, kein Blocken — zählt trotzdem als Befund).
- **D4 — Betroffene 9 Hooks:** check-version-sync, typecheck-gate, test-gate, release-gate
  (PreToolUse Bash|PowerShell) · secrets-guard, compose-guard (PreToolUse Edit|Write) ·
  run-matching-test, json-guard (PostToolUse) · stop-verify (Stop).
- **D5 — `.gitignore`:** `.claude/hooks/hook-log.jsonl` (Telemetrie ist lokal, nie committen).
- **D6 — Tests:** neuer Smoke-Test `test/hooks/hook-log.test.js` für den Helper (schreibt Zeile
  wenn Zielordner existiert; wirft nie bei fehlendem Ordner; JSONL-parsebar). Die bestehenden
  Hook-Tests bleiben unverändert grün (Fixture-cwds → silent skip beweist D2 nebenbei).
  End-to-End-Verify (Prompt): absichtlich kaputter JSON-Edit an `.homeychangelog.json` →
  live json-guard meldet → block-Zeile in hook-log.jsonl nachweisen → Edit rückgängig.
- **D7 — Dedupe** `.claude/hooks/lib/spawn-env.js`: exportiert `spawnEnv()` (Kopie von
  `process.env` ohne `NODE_TEST_CONTEXT`/`NODE_TEST_WORKER_ID`); test-gate, stop-verify und
  run-matching-test nutzen ihn (Verhalten durch bestehende Tests abgedeckt — reiner Refactor).

## §4 Triage-Routine — Entscheidungen

- **D8 — Mechanismus:** lokaler Scheduled Task (B1), `taskId: violet-nightly-triage`, Cron
  `30 3 * * *` (Lokalzeit; nach dem CI-Nightly 02:17, dessen Ergebnis die Triage sichtet).
- **D9 — Selbstständiger Task-Prompt** (jeder Lauf startet ohne Session-Gedächtnis): Repo-Pfad,
  Schrittfolge (npm test · npx homey app validate --level publish · Versions-Sync app.json vs
  .homeycompose/app.json · letzter CI-Lauf via gh · offene Follow-ups aus MEMORY.md/project-status
  gegenchecken), Inbox-Format, Commit-Anweisung (nur `docs/dashboard/triage-inbox.md`, Prefix
  `chore(triage):`), Verhaltensregeln (read-only außer Inbox-Commit; keine Fixes, nur Findings).
- **D10 — Inbox-Format** (`docs/dashboard/triage-inbox.md`): Kopf mit Stand-Datum; Abschnitte
  `## Offen` / `## Neu (<Datum>)` / `## Erledigt (<Datum>)`; jeder Eintrag eine Zeile mit Datum,
  Quelle (Test/validate/CI/Follow-up) und Kurzbefund. Die Datei wird von der Routine gepflegt
  (Einträge wandern zwischen Abschnitten), nicht endlos angehängt.
- **D11 — Manueller Testlauf ZUERST** (Prompt-Auflage): die D9-Schritte einmal in dieser Session
  ausführen, Inbox erzeugen + committen; erst danach den Task anlegen (Kadenz setzen).
- **D12 — §7-Regel** (CLAUDE.md, Rules-Block): „Jede neue Milestone-Session und der
  release-readiness-Subagent lesen `docs/dashboard/triage-inbox.md` ZUERST (falls vorhanden)."
  Ergänzend eine Zeile im release-readiness-Agent (der die Regel operativ umsetzt).

## §5 milestone-checkpoint-Erweiterungen — Entscheidungen

- **D13 — Memory-Konsolidierung** als neuer Schritt NACH der Workflow-Retro (wird Schritt 5;
  Dashboard-Update rückt auf 6, Handover wird 7). Inhalt (Dreaming-Muster): Sessions seit dem
  letzten Checkpoint sichten (`search_session_transcripts`, falls verfügbar — sonst Dashboard-
  `log[]` + git log als Quelle); `memory/`-Dateien deduplizieren/kürzen (Erledigtes eindampfen —
  `project-status` trägt noch die volle M2/M3-Detailhistorie); Widersprüche auflösen (z. B.
  veraltete „ab nächster Session"-Aussagen). ERGEBNIS IMMER ALS DIFF ZUM REVIEW, NIE direkt
  anwenden; offene Follow-ups und Security-Notizen NIE löschen.
- **D14 — Handover** als finaler Schritt 7: nächsten `status: "todo"`-Milestone (erster in
  Dashboard-Reihenfolge) lesen, Resume-Prompt als Push-Benachrichtigung senden (PushNotification;
  Hinweis auf `/remote-control <id>`; Prompt-Text gekürzt auf Titel + erste Zeilen — Push hat
  Längenlimits, der volle Prompt steht im Dashboard), danach fragen, ob der Milestone direkt in
  dieser Session gestartet werden soll. Kein Push-Kanal verfügbar → Prompt im Chat zeigen und
  das vermerken.
- **D15 — Retro-Signalquelle:** Schritt-4-Punkt 1 („Signal sammeln") liest zusätzlich
