# M5.9 — package-guard (Slopsquatting Defense): Design Spec

- **Date:** 2026-07-14
- **Status:** Approved design (brainstorming 2026-07-14) — feeds the M5.9 implementation plan
- **Threat model (binding):** `docs/superpowers/security/2026-07-14-m5.9-package-guard-threat-model.md` (SR-01…SR-10)
- **Approach decision:** Approach A — one hook, direct registry HTTPS (vs. B: split hooks; C: `npm view` child process). Chosen for: single policy in one file, pure logic testable offline, no child-process latency. Explicitly traded away: `.npmrc` private-registry inheritance (not needed — public npm only).
- **User decisions (2026-07-14):** all three layers in scope · existence **+ heuristics** policy · **fail closed** on unverifiable · dashboard entry **M5.9**, tooling-only (no version bump, M4.5 precedent).

## 1. Problem

LLM-generated code and plans hallucinate npm package names (~20% of recommendations; 43% of hallucinations repeat across runs). Attackers pre-register these names ("slopsquatting") so that a later `npm install` executes attacker code with developer privileges. Prompt-level care lowers the rate but cannot close the hole — the protection must be a deterministic gate at the install/manifest boundary (CLAUDE.md §10 layer 1: hooks = "must NEVER happen").

## 2. Defense layers (all in scope)

| Layer | What | Where |
|---|---|---|
| **1 — Guard hook** | PreToolUse `package-guard.js` blocks unverified package names in install commands and `package.json` edits | `.claude/hooks/` + `.claude/settings.json` |
| **2 — Plan constraint** | New dependencies require a recorded existence proof in specs/plans | CLAUDE.md "Dependency Hygiene" block |
| **3 — Install hardening** | `--ignore-scripts` convention; CI: `npm ci --ignore-scripts` + `npm audit` | CLAUDE.md block + `.github/workflows/ci.yml` |

## 3. Layer 1 — hook architecture

### 3.1 Interception (SR-03)

Registered under **both** existing PreToolUse matcher groups, branching on `tool_name`:

- **`Bash` / `PowerShell` — command path:** scan `tool_input.command` for install invocations: `npm install|i|add|exec`, `npx`, `yarn add`, `pnpm add`. The parser walks each segment of compound commands (`&&`, `;`, `|`), skips option tokens (`-…`/`--…`), and collects package-spec tokens. Bare `npm install`/`npm ci` (no specs) → no registry names → pass without queries.
- **`Edit` / `Write` — manifest path:** fires only when `file_path` ends with `package.json` (any directory, `node_modules` excluded). Post-edit content: `Write` → `tool_input.content`; `Edit` → apply `old_string → new_string` (respecting `replace_all`) to the **pre-edit disk file**. Parse pre and post JSON; diff the dependency blocks `dependencies`, `devDependencies`, `optionalDependencies`, `peerDependencies`, `overrides` (recursive). Only **newly introduced or changed-spec** entries are verified — unchanged existing deps are never re-blocked (SR-01 no-self-DoS). If the post-edit content is not parseable JSON or `old_string` is not found, the guard passes (the edit/`json-guard` fails on its own).

### 3.2 Name resolution (SR-05)

For each candidate spec, resolve the **registry name actually fetched**:

- plain: `lodash`, `@scope/name`, `name@^1.2.3` → name part
- alias: `"key": "npm:real-name@range"` → `real-name` (the key is NOT verified)
- skipped (not registry-squattable, visually auditable): `git:`/`git+…`/`github:user/repo`, `file:`, `link:`, `workspace:`, `http(s)://` tarball URLs

### 3.3 Verdict rules

| Rule id | Condition | Exit |
|---|---|---|
| `not-found` | `GET registry.npmjs.org/<name>` → 404 | **2 (block)** |
| `too-new` | `time.created` < **90 days** ago (`MIN_AGE_DAYS`) | **2** |
| `low-adoption` | `api.npmjs.org/downloads/point/last-week/<name>` < **500** (`MIN_WEEKLY_DOWNLOADS`) | **2** |
| `verify-unavailable` | network error / timeout (**5 s** per request, `REQUEST_TIMEOUT_MS`) | **2 — fail closed** (SR-04) |
| `runtime-dep` | new entry in `dependencies` while `BLOCK_RUNTIME_DEPS = true` (Violet: `dependencies` stays `{}`) | **2** |
| — pass | name exists, ≥ 90 days old, ≥ 500 weekly downloads | 0 |

- Thresholds are named constants at the top of `package-guard.js`, documented in the file header.
- **Failure-mode asymmetry (deliberate, documented in header):** harness-input parse errors fail **open** (stdin comes from the trusted harness — house convention, cf. secrets-guard); only the **network verification** path fails closed (a gate an attacker can disable via timeout is not a gate).
- Block message (stderr): package name, rule id, one-line reason, and the legitimate override — *"if intentional, install it manually in your own terminal"* (human installs bypass hooks by design).
- Every verdict logged via existing `logHook('package-guard', 'pass'|'block', cwd)` (SR-10).
- **SR-08 deviation:** the threat model mentioned a brief per-name cache; dropped as YAGNI — only newly-introduced names are ever queried, so query volume is already minimal. SR-08's metadata-only core stands: the guard never downloads or executes a candidate package.

### 3.4 Components

| File | Responsibility |
|---|---|
| `.claude/hooks/lib/package-specs.js` | **Pure, no IO** (JSDoc per `/documenting-code`): `parseInstallCommand(command) → string[]` (specs) · `resolveSpecName(keyOrName, spec) → string\|null` (null = skip) · `diffNewDeps(preJson, postJson) → {name, spec, depBlock}[]` (`depBlock` = which dependency section it came from) · `verdict({exists, createdAt, weeklyDownloads}, now, opts) → {ok: true} \| {ok: false, rule}` |
| `.claude/hooks/package-guard.js` | Thin shell: stdin JSON, tool branching, pre-edit disk read, `https.get` to registry/downloads API (base URLs overridable via `PACKAGE_GUARD_REGISTRY_BASE` / `PACKAGE_GUARD_DOWNLOADS_BASE` env — test seam), exit codes |
| `test/hooks/package-guard.test.js` | Unit tests of the pure lib (offline) + spawn tests of the hook against a local `node:http` stub registry on an ephemeral port. **No real network in tests.** |
| `.claude/settings.json` | `node .claude/hooks/package-guard.js` appended to both matcher groups |

## 4. Layer 2 — CLAUDE.md "Dependency Hygiene" block

Short rule block appended to §5 (Security by Design) area, generic wording:

> **Dependency Hygiene:** No new dependency (runtime or dev, incl. `npm:` alias targets) enters a spec/plan without an existence proof — a registry lookup or context7 doc hit — recorded in the plan's Tech Stack/Global Constraints. Agent-run installs use `--ignore-scripts`. The `package-guard` hook enforces registry existence + freshness/adoption at install/manifest time; its override path is a human terminal install.

## 5. Layer 3 — CI hardening

`.github/workflows/ci.yml`: `npm ci` → **`npm ci --ignore-scripts`** (current devDeps need no lifecycle scripts — verified in plan), plus **`npm audit`** as an additional **blocking** step (the dep tree is tiny and currently clean; a failing audit should fail CI).

## 7. Testing strategy (TDD)

- Pure-lib unit tests first: command parsing (incl. compound commands, flags, scopes, aliases), dep-diffing (all five blocks, `overrides` recursion), verdict thresholds (boundary: exactly 90 days / exactly 500 downloads → **pass**; just under → block).
- Hook spawn tests: stub registry serving crafted metadata → each rule id observed on exit 2 + stderr contains name and rule; pass cases exit 0; stub down (closed port) → `verify-unavailable`; non-`package.json` edits and non-install commands → exit 0 with **zero** requests (stub asserts no hits).
- Existing suites (`npm test`, `npm run typecheck`) stay green; hook must not slow non-install tool calls measurably (early-exit before any IO).

## 8. Out of scope / accepted (from threat model §6)

Established-but-compromised packages (CI audit partially covers) · package content scanning · non-npm ecosystems · human terminal installs (= override path) · git/file/URL specs (skipped, auditable) · fail-closed loop friction on registry outages (accepted by decision).
