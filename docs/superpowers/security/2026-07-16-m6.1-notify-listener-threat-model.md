# Violet Homey App — M6.1 Inbound NOTIFY Listener: Threat Model & Security Requirements

- **Date:** 2026-07-16
- **Status:** Draft — feeds the M6.1 plan (written before code, per CLAUDE.md §5)
- **Method:** STRIDE threat model → traceable security requirements (`security-requirement-extraction` skill)
- **Scope:** Milestone **M6.1 — Inbound HTTP alarm notifications**: embedded unauthenticated HTTP listener (`lib/NotifyServer.js`, default port 22222) inside the Pool device, turning Violet NOTIFY requests (`?ERRORCODE&SUBJECT`) into the `alarm_received` device Flow trigger.
- **References:** design spec `docs/superpowers/specs/2026-06-26-violet-http-notifications-design.md` (§3, §7, §10); M3 threat model (pattern); `.claude/agents/write-path-security-reviewer.md` (its scope explicitly covers this listener)

> The M3/M5.7/M5.9 surfaces were *outbound* or tooling. This is the app's **first inbound network listener** — a new class of attack surface: anything on the LAN can talk to it, and the Violet cannot authenticate itself (plain HTTP, no auth — controller limitation).

---

## 1. Assets & confirmed facts

| Asset | Why it matters |
|---|---|
| **App process availability** | The listener runs inside the same app that does pool monitoring, the LSI safety net, and (interlocked) write control. A listener crash or memory exhaustion takes down **all** of it — a regression this app would not have without M6.1. |
| **Flow-trigger integrity** | Users will wire `alarm_received` to real actions (push notifications, sirens, maybe equipment logic). Forged or flooded alarms cause false alerts and Flow spam. |
| **Homey Pro resources** | Constrained device; unbounded request buffering or per-request leaks are a real hazard. |
| **Log integrity** | `SUBJECT` is attacker-controllable free text; unsanitized it can inject fake lines into logs/diagnostics. |

**Inbound request (confirmed from manual p.153 + user's live setup):**
```
GET http://<homey>:<port>/<any-path>?ERRORCODE=<code>&SUBJECT=<text>   (POST variant also possible)
```
No authentication, no TLS — controller limitation, identical to the Micro Web Server status quo being replaced.

## 2. Trust boundaries

```
[Violet]──①──►(LAN, plain HTTP, no auth)──②──►[NotifyServer :22222]──③──►[parseAlarm]──►[PoolDevice._fireAlarm]──④──►[user Flows]
  sends alarms      any LAN device can send        accepts sockets        pure parsing        trigger + tokens
```

- **① / ② LAN → listener:** the listener cannot distinguish the Violet from any other LAN device. Everything arriving here is **untrusted input**.
- **③ listener → parseAlarm → device:** the only gate where input is validated, bounded, and sanitized.
- **④ trigger → Flows:** tokens become user-visible/actionable; whatever crosses ③ lands in the user's automation.

## 3. Threats (STRIDE)

| ID | STRIDE | Threat | Impact | Likelihood | Mitigating reqs |
|---|---|---|---|---|---|
| **T-M6-S1** | Spoofing | Any LAN device forges NOTIFY requests → false `alarm_received` triggers (false alerts, panic actions in user Flows) | Misleading automation | Medium (trivial once on LAN) | SR-04, SR-05, SR-08 |
| **T-M6-T1** | Tampering | Malicious query values: oversized strings, malformed %-encodings (`decodeURIComponent` throws), control characters/newlines → crash, log injection, or garbage tokens in Flows | App crash / log forgery | Medium | SR-01, SR-02, SR-03 |
| **T-M6-D1** | DoS | Request flood, slowloris, or unbounded POST bodies exhaust sockets/memory or block the event loop → whole app (monitoring + LSI net) down | Availability of all app functions | Medium | SR-01, SR-02, SR-06 |
| **T-M6-D2** | DoS | Alarm flood fires thousands of Flow triggers → Homey Flow/timeline spam, resource drain downstream | Homey-wide nuisance/load | Medium | SR-06 |
| **T-M6-D3** | DoS | Port conflict (`EADDRINUSE`, e.g. Micro Web Server still on the old port) or bind error breaks device init | Device unusable | Medium | SR-07 |
| **T-M6-I1** | Info disclosure | Listener responses echo input or leak internals (stack traces, versions, readable endpoints) | Recon / reflected content | Low | SR-09 |
| **T-M6-E1** | Elevation | Parsed alarm data reaches control/write logic (directly or via later refactor) — an unauthenticated LAN request would then drive equipment | Physical safety | Low (design forbids it) — severe if it happens | SR-04 |
| **T-M6-R1** | Repudiation | No record of what fired a trigger when diagnosing false alarms | Slower incident analysis | Low | SR-08 |

## 4. Security requirements

Each is traceable (→ threat), testable, and prioritized. `req_type` ∈ {Functional, Non-functional, Constraint}. ASVS = OWASP ASVS control family.

| ID | Title | Type | Priority | Threats | ASVS |
|---|---|---|---|---|---|
| **SR-M6-01** | Listener never crashes the app: every conceivable request (malformed method/URL/encoding, oversized, aborted, slow) is handled without an uncaught throw | Non-functional | **Critical** | T1, D1 | V5, V11 |
| **SR-M6-02** | Hard input bounds: request-URL length cap, POST body read cap with abort, per-socket timeout | Functional | High | T1, D1 | V5, V11 |
| **SR-M6-03** | Token validation & sanitization: `ERRORCODE` matches `^[A-Za-z0-9]{1,8}$`; `SUBJECT` length-capped and stripped of control chars/newlines before tokens and logs; malformed %-encoding → treated as invalid, never thrown | Functional | High | T1 | V5 |
| **SR-M6-04** | Trigger-only data path: parsed alarm data feeds **only** the Flow trigger tokens and a log line — never capability writes, settings, WriteClient, or any control logic | Constraint | **Critical** | S1, E1 | V4 |
| **SR-M6-05** | Unauthenticated LAN exposure is an explicit, documented accepted risk (controller limitation; parity with the replaced Micro Web Server), stated in README/store notes with "LAN only, do not port-forward" guidance | Constraint | Medium | S1 | V1 |
| **SR-M6-06** | Trigger-rate limit: bounded number of trigger fires per time window (excess dropped with a single warning log), so a flood cannot spam Flows | Functional | Medium | D1, D2 | V11 |
| **SR-M6-07** | Robust lifecycle: bind errors (`EADDRINUSE` et al.) surface via `this.error` and leave the device fully functional (polling unaffected); port-change rebinds close the old server first; `onUninit` closes cleanly (singleton guard) | Functional | High | D3 | V11 |
| **SR-M6-08** | Received alarms logged (sanitized code + truncated subject + source IP), so trigger causes are auditable | Functional | Low | R1, S1 | V7 |
| **SR-M6-09** | Minimal static responses: fixed `200 OK`/`400 Bad Request` bodies; no echo of request content, no stack traces, no version info, no readable endpoints | Non-functional | Medium | I1 | V14 |

### Expanded — the two CRITICALs

**SR-M6-01 — Listener never crashes the app**
*As a pool owner, I need the alarm listener to survive any garbage the LAN throws at it, so that a stray packet can never take down pool monitoring or the LSI safety net.*
- Acceptance criteria:
  - [ ] `parseAlarm` is total: for **any** (method, url, body) input it returns `{ errorcode, subject }` or `null` — it never throws (including malformed `%`-sequences, non-UTF-8 bytes, absurd lengths).
  - [ ] The server registers `error` handlers on both server and sockets; request-handler body is wrapped so no code path can reject/throw uncaught.
  - [ ] Oversized/slow requests are aborted (SR-M6-02 limits), not buffered indefinitely.
- Test cases:
  - Fuzz-style unit table: `%zz` bad encoding, 100 KB URL, empty request, binary junk body, missing params, duplicate params → `null` or parsed result, never a throw.
  - Integration test: request with immediately-destroyed socket does not emit an unhandled error.

**SR-M6-04 — Trigger-only data path**
*As a pool owner, I need inbound alarm data to be strictly read-only display data, so that an unauthenticated LAN request can never drive pool equipment.*
- Acceptance criteria:
  - [ ] The only consumers of `parseAlarm` output are the `alarm_received` trigger tokens/state and the SR-M6-08 log line.
  - [ ] No import/require path connects `NotifyServer` output to `WriteClient`, capability setters, or settings.
  - [ ] `write-path-security-reviewer` subagent review of the M6.1 diff confirms this before merge.
- Test cases:
  - Grep/review assertion: `drivers/pool/device.js` alarm handler calls trigger + log only.
  - Unit test: `_fireAlarm` performs no capability/setting mutation.

## 6. Accepted risks / out of scope

- **No authentication/TLS on the listener** (SR-M6-05): the Violet can send neither; requiring them would break the feature. Residual risk — anyone on the LAN can fake or suppress-by-flooding alarms — is **accepted by design** and documented. Parity note: the replaced Micro Web Server setup had exactly the same exposure, plus a second app's attack surface on top.
- **Source-IP pinning to the paired Violet host:** considered, **rejected for now** — the Violet's source IP may differ from the paired hostname (DHCP, DNS vs IP), and false negatives would silently kill the feature. SR-M6-08's logged source IP gives the diagnostic value without the fragility. Revisit only on real-world abuse evidence.
- **Network-level DoS** (SYN floods etc.): out of scope — home LAN, OS concern. In-process resource exhaustion **is** in scope (SR-M6-01/02/06).
- **No PII/GDPR scope:** alarm codes and subjects are appliance telemetry, no personal data; nothing is stored beyond logs.
- **Binding to `0.0.0.0` (all interfaces) is a conscious choice:** on a Homey Pro every interface is LAN (no separate untrusted/public NIC to isolate from), so binding to a single interface would add complexity without narrowing the exposure. Revisit if the platform ever adds VPN or multi-homing support that puts a non-LAN interface on the same host.
