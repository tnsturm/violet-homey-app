# Violet Homey App — M5.9 Package Guard: Threat Model & Security Requirements

- **Date:** 2026-07-14
- **Status:** Draft — feeds M5.9 brainstorming/spec (not yet implemented)
- **Method:** STRIDE threat model → traceable security requirements (`security-requirement-extraction` skill)
- **Scope:** Milestone **M5.9 — package-guard (slopsquatting defense)**: protecting the agentic development loop from hallucinated/attacker-registered npm package names ("slopsquatting", CSO Online 2025-04). Three layers: (1) deterministic PreToolUse guard hook, (2) plan-constraint requiring existence proof for new dependencies, (3) install hardening (`npm ci`, `--ignore-scripts`, CI audit).
- **References:** CLAUDE.md §5/§10; `secrets-guard.js` (hook pattern); M3 threat model (doc pattern); research: ~20% of LLM-recommended packages are fake, 43% of hallucinations repeat across runs (arXiv 2406.10279 via CSO Online)

> This document exists so the security-relevant decisions for the package-install path are made **before** code, per CLAUDE.md §5. Carry every requirement below into the M5.9 plan as a verifiable step.

## 1. Assets & confirmed facts

| Asset | Why it matters |
|---|---|
| **Developer workstation** | `npm install` runs arbitrary lifecycle scripts (preinstall/postinstall) with the developer's privileges. A single malicious package = full workstation compromise. |
| **Credentials reachable from the workstation** | Homey CLI session token, GitHub push access (`gh` auth), `VIOLET_WRITE_PASSWORD` env var (used by secrets-guard), SSH keys. A postinstall script can exfiltrate all of these. |
| **Repo / published-app integrity** | `dependencies` is deliberately `{}` — runtime deps would be bundled into the published Homey app and reach **end users' Homeys**. devDependencies stay local/CI but still execute on install. |
| **CI runner (GitHub Actions, M4.6)** | Runs `npm` installs with a `GITHUB_TOKEN`; a malicious dep compromises CI and anything the token can touch. |
| **`package.json` / `package-lock.json`** | The manifests are the *latent* attack vector: an unverified name written today is installed later by `npm install`/`npm ci` (human, agent, or CI) without re-scrutiny. |

**Confirmed facts:**
- All three `devDependencies` use or coexist with the **`npm:` alias pattern** (`"@types/homey": "npm:homey-apps-sdk-v3-types@^0.3.12"`) — the *alias target* is the real registry name and must be verified, not the key.
- Claude Code hooks only intercept **agent** tool calls; a human running `npm install` in a terminal bypasses the hook by design (that IS the override path).
- Hook verdicts are deterministic and apply in every permission mode, incl. Auto Mode (CLAUDE.md §10).

## 2. Trust boundaries

```
[LLM training data / plan docs]──①──►[executing agent]──②──►[npm CLI]──③──►(npm registry)
     hallucinated names may           proposes install       resolves name;    anyone can register
     be embedded in plans             or manifest edit       runs lifecycle    any unclaimed name
                                            │                scripts ④
                                            └──②'──►[package.json edit]──(later install by human/CI)
```

- **① plans/model → agent:** package names originating from model output are **untrusted data**, even when they appear in our own committed plan documents (writing-plans output is model-generated).
- **② agent → npm CLI (Bash/PowerShell):** the moment of no return — lifecycle scripts execute during install. The guard must sit here.
- **②' agent → manifest edit:** no code runs yet, but the name becomes trusted-looking repo content that a later install resolves. The guard must sit here too.
- **③ npm CLI → registry:** the registry does not vouch for names — an attacker can pre-register any hallucinated name (the slopsquatting mechanism).
- **④ install → workstation:** lifecycle scripts run with full user privileges unless `--ignore-scripts`.

## 3. Threats (STRIDE)

| ID | STRIDE | Threat | Impact | Likelihood | Mitigating reqs |
|---|---|---|---|---|---|
| **T-M59-S1** | Spoofing | Agent installs a hallucinated name an attacker has **pre-registered** on the registry (slopsquat) | Workstation + credential compromise | Medium (43% of hallucinations are repeatable → attackers can farm them) | SR-01, SR-02, SR-07 |
| **T-M59-S2** | Spoofing | Hallucinated **`npm:` alias target** (key looks legit, target is fake/squatted) or hallucinated name in `overrides`/`resolutions` | Same as S1, hidden from casual review | Low–Med | SR-05, SR-03 |
| **T-M59-T1** | Tampering | Malicious **lifecycle script** modifies workstation, repo, or other packages during install | Full compromise; persistent backdoor | Med (given S1 occurs) | SR-07 |
| **T-M59-T2** | Tampering | Unverified name written to `package.json` today is installed **later** by human/CI without scrutiny (latent injection) | Deferred S1 with higher trust | Medium | SR-03, SR-09 |
| **T-M59-S3** | Spoofing (guard bypass) | Agent reaches the registry via an uncovered vector: `npm i`/`add`, `npx <pkg>`, `yarn`/`pnpm`, `npm exec` | Guard silently skipped | Medium | SR-03 |
| **T-M59-I1** | Info disclosure | Installed malicious package exfiltrates env vars / tokens (`VIOLET_WRITE_PASSWORD`, `GITHUB_TOKEN`, Homey session) | Controller + repo + CI takeover | Med (standard payload) | SR-01, SR-02, SR-07 |
| **T-M59-E1** | Elevation | A malicious **runtime** dependency gets bundled into the published app → end users' Homeys | Supply-chain hit on app users | Low (`dependencies` frozen at `{}`) | SR-06, SR-03 |
| **T-M59-D1** | DoS (of the loop) | Attacker or outage makes the registry unreachable → fail-closed guard blocks legitimate installs | Loop friction, no compromise | Low | SR-04 (accepted cost), SR-08 |
| **T-M59-R1** | Repudiation | No record of what the guard allowed/blocked when diagnosing an incident | Slower incident analysis | Low | SR-10 |

## 4. Security requirements

Each is traceable (→ threat), testable, and prioritized. `req_type` ∈ {Functional, Non-functional, Constraint}. ASVS = OWASP ASVS control family.

| ID | Title | Type | Priority | Threats | ASVS |
|---|---|---|---|---|---|
| **SR-01** | Every package name in an intercepted install command or manifest edit must **exist** on the npm registry before the action proceeds | Functional | **Critical** | S1, I1 | V14 |
| **SR-02** | Existing packages failing **freshness/adoption heuristics** (first publish < 90 days ago OR < 500 weekly downloads) are blocked; override = human installs manually | Functional | **Critical** | S1, I1 | V14 |
| **SR-03** | Guard intercepts **all agent install/manifest vectors**: `npm install|i|add|exec`, `npx <pkg>`, `yarn add`, `pnpm add` (Bash *and* PowerShell tool), and Edit/Write to `package.json` (all dep blocks incl. `overrides`) | Functional | High | S3, T2, E1 | V14 |
| **SR-04** | **Fail closed** when the registry is unreachable/times out — block with a distinct "could not verify" message (never silently allow) | Constraint | High | S1, D1 | V14 |
| **SR-05** | `npm:` **alias targets** (and spec strings generally) are parsed and the *resolved registry name* is verified, not the manifest key | Functional | High | S2 | V14 |
| **SR-06** | Guard warns/blocks on any addition to `dependencies` (must stay `{}` per project constraint) — runtime deps are a publish-time supply-chain surface | Functional | Medium | E1 | V14 |
| **SR-07** | Install hardening (Layer 3): agent installs use `--ignore-scripts`; CI installs use `npm ci` from the committed lockfile + an audit step | Constraint | High | T1, S1, I1 | V14 |
| **SR-08** | Verification is **metadata-only** (registry HTTP GET / `npm view`) — the guard never downloads or executes the candidate package; per-name result cached briefly to keep the loop fast | Non-functional | Medium | D1 | V14 |
| **SR-09** | Plan constraint (Layer 2): no new dependency enters a plan without an existence proof (registry lookup or context7 doc hit) recorded in the plan's Tech Stack/Global Constraints | Constraint | Medium | T2 | — |
| **SR-10** | Guard logs every verdict (pass/block + rule id) via the existing `logHook`; block messages name the package, the failed rule, and the legitimate override path | Functional | Low | R1 | V7 |

### Expanded — the two CRITICALs

**SR-01 — Existence check before any install/manifest write**
*As the operator of an agentic dev loop, I need every package name an agent tries to install or persist verified against the real registry, so that a hallucinated name can never resolve to attacker code.*
- Acceptance criteria:
  - [ ] `npm install <nonexistent-name>` issued by the agent is blocked (exit 2) with the name and rule id in the message.
  - [ ] An Edit/Write introducing a nonexistent name into any dependency block of `package.json` is blocked.
  - [ ] Scoped names (`@scope/name`), version specs (`name@^1.2.3`), and git/file/URL specs are parsed correctly (git/file/URL specs are out of registry scope → see §6).
  - [ ] Names already present in the committed `package.json`+lockfile are not re-blocked (no self-DoS on `npm ci`).
- Test cases:
  - Test: install command with a known-fake name → exit 2, message contains name + `not-found`.
  - Test: install of an existing, old, popular package (e.g. `typescript`) → exit 0.
  - Test: `npm ci` (no names) → exit 0 without registry queries.

**SR-02 — Freshness/adoption heuristics (the actual slopsquat catch)**
*As the operator, I need names that resolve but look like freshly-registered squats blocked, so that an attacker pre-registering hallucinated names gains nothing.*
- Acceptance criteria:
  - [ ] Package with first publish < 90 days ago → blocked with rule id `too-new`.
  - [ ] Package with < 500 weekly downloads → blocked with rule id `low-adoption`.
  - [ ] Block message states the override explicitly: "if intentional, install manually in your own terminal".
  - [ ] Thresholds live in one place (top-of-file constants), documented in the hook header.
- Test cases:
  - Test: metadata stub with `created` 10 days ago → blocked (`too-new`).
  - Test: metadata stub with 12 weekly downloads → blocked (`low-adoption`).
  - Test: both heuristics pass → allowed.

## 6. Accepted risks / out of scope

- **Established malicious packages pass the heuristics** (old + popular but compromised, e.g. a hijacked maintainer account): out of scope for this guard — partially covered by the CI `npm audit` step (SR-07) and lockfile pinning; full content scanning is a different tool class.
- **Content is never inspected** (SR-08 is metadata-only by design): the guard verifies *identity and provenance signals*, not code. Accepted.
- **Human terminal installs bypass the hook**: by design — the human IS the override path for false positives (genuinely new packages). The block message directs there.
- **git/file/URL dependency specs** (`github:user/repo`, `file:../local`): no registry to verify against. The guard blocks registry-name resolution failures but *allows* explicit git/file/URL specs — they are visually auditable and not registrable by squatters. Documented, accepted.
- **Non-npm ecosystems** (PyPI etc.): out of scope — this project is npm-only. The framework-repo mirror notes the pattern generalizes.
- **Fail-closed loop friction** (T-M59-D1): accepted per explicit user decision 2026-07-14 — a security gate an attacker can disable via timeout is not a gate.
- **Hook-input parse failure fails open** (matches every existing hook): the harness is a *trusted* input source; only the *network* verification path fails closed. This distinction is deliberate and documented in the hook header. **Refinement (code review 2026-07-14):** only the stdin *envelope* is harness-trusted; `tool_input` content is agent-authored (prompt-injectable). A guard crash during a guard-relevant action (install command / package.json edit) therefore fails **closed** (`guard-error`); the overrides-flatten is iterative so npm-valid-but-hostile-deep nesting cannot crash-bypass the analysis.
- **Variable-indirection installs bypass static parsing** (e.g. `PKG=x && npm install $PKG`): the spec token resolves to no verifiable name and passes. Accepted — the guard polices an honest-but-hallucinating agent (the slopsquatting mechanism); a deliberately evasive agent has arbitrarily many alternative channels (script files, `node -e`, `curl | sh`) that no command parser can close. Defense against that class is Layer 3 (`--ignore-scripts`, lockfile-only CI installs, audit) plus the human in the loop.
- **Large registry documents make one-off verifications slow** (full metadata of e.g. `typescript` is tens of MB): accepted — the request timeout is inactivity-based, so a slow-but-flowing download does not false-block; new-dependency events are rare by design (SR-09 makes them plan-level decisions).
