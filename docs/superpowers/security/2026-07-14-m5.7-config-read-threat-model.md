# Violet Homey App — M5.7 Config-Autoconfig: Threat-Model-Delta (getConfig-Lesepfad)

- **Date:** 2026-07-14
- **Status:** Delta zum M3-Threat-Model (`2026-06-30-m3-write-control-threat-model.md`) — feeds M5.7 Spec/Plan
- **Method:** STRIDE-Delta → traceable Requirements (`security-requirement-extraction` skill)
- **Scope:** Neuer **Lesepfad** `GET http://<host>/getConfig?<key,key,prefix…>` als Zweitquelle der
  Feature-Detection; kein neuer Schreibpfad.
- **Live verifiziert (2026-07-14, Referenzgerät FW 1.2.1):** gezielte Key-/Präfix-Queries sind
  **ohne** Basic Auth lesbar; `?ALL` → 401; Secret-Keys einzeln → Klartext `Access restricted, no
  Auth found` (kein JSON); Secret-Keys in gemischten Queries werden still gedroppt;
  `SYSTEM_allow_client_ips_without_auth` ist leer (Firmware-Default, keine IP-Freigabe).

## 1. Delta-Assets

| Asset | Delta |
|---|---|
| Gespeicherte Credentials (M3: Store/Settings) | Werden **konditional** wiederverwendet: nur wenn eine gezielte Query 401/`Access restricted` liefert (ältere/strikter konfigurierte Firmware). Default-Pfad ist credential-frei. |
| Detection-Autorität (Capability-Menge) | **Neu:** getConfig-Antworten entscheiden mit, welche Capabilities existieren (Cover: negativ autoritativ). Spoofte Antworten können Sichtbarkeit (Tiles/Alarme) verändern — nicht aber Schreibaktionen auslösen. |
| Secrets/PII in `getConfig?ALL` | Werden **nie angefragt** (Whitelist-Queries statt `?ALL`) — Hazard nur bei Code-Drift. |

## 2. Delta-Threats (STRIDE)

| ID | STRIDE | Threat | Impact | Likelihood | Reqs |
|---|---|---|---|---|---|
| **T-M57-I1** | Info disclosure | Code-Drift zu `?ALL`/breiter Whitelist lässt Secrets (WLAN/SMTP/PIN) in App-Speicher/Logs gelangen | Credential-/PII-Leak | Low (nur durch eigenen Code) | SR-11, SR-12 |
| **T-M57-I2** | Info disclosure | 401-Fallback sendet Basic-Auth-Header über Klartext-HTTP auf dem Lesepfad | Credential-Sniffing (wie M3 T-M3-I1) | Med | SR-14 (= SR-01/02/03 auf Lesepfad ausgedehnt) |
| **T-M57-S1** | Spoofing | Rogue Host/MITM liefert manipulierte Config → Detection versteckt Tiles/Alarme (z. B. Dosier-Alarm) | Verlust von Monitoring-Sichtbarkeit; keine physische Aktion | Low–Med | SR-15, SR-08 (Host-Pinning gilt) |
| **T-M57-T1** | Tampering/Robustheit | Nicht-JSON-Antwort (`Access restricted`-Klartext), Partial-/Mischtyp-Antwort (String `"0"` vs. Number `0`) bricht Parser oder vergiftet den Store | Detection-Ausfall/Fehlklassifikation | Med (live belegt) | SR-13 |
| **T-M57-D1** | DoS (eigenverschuldet) | Config-Fetch-Fehler reißen die Geräteverfügbarkeit mit runter | Fälschlich „unavailable" | Low | SR-16 |

## 3. Delta-Requirements

| ID | Titel | Typ | Prio | Threats |
|---|---|---|---|---|
| **SR-11** | getConfig-Queries nur aus **einer** Whitelist-Konstante (Keys/Präfixe); `?ALL` existiert im Config-Client nicht | Constraint | High | I1 |
| **SR-12** | Rohe getConfig-Antwort nie loggen/persistieren; nur whitelisted Keys überleben den Parser in den Device-Store | Non-functional | **Critical** | I1 |
| **SR-13** | Parser fail-soft: Nicht-JSON/Partial/Mischtypen → letzter guter Facts-Stand bleibt; nie Exception bis device.js | Functional | High | T1 |
| **SR-14** | Credentials auf dem Lesepfad nur im `Authorization`-Header, nur nach 401/`Access restricted`-Signal; SR-01/SR-02 (nie in URL, nie in Logs) gelten unverändert | Constraint | High | I2 |
| **SR-15** | Config-getriebene Capability-**Entfernung** nur für explizit negativ-autoritative Gruppen (M5.7: nur Cover); `force`-Override gewinnt immer | Functional | Medium | S1 |
| **SR-16** | Config-Fetch-Fehler beeinflussen die Geräteverfügbarkeit nicht (Availability gehört dem getReadings-Pfad); Logging gedrosselt (1. Fehler Warnung, dann ≤1/5 min, Recovery-Info) | Non-functional | Medium | D1 |

### Akzeptanzkriterien (testbar, → Plan/TDD)

- **SR-11:** Unit-Test: gebaute URL enthält ausschließlich Whitelist-Einträge; kein Vorkommen von
  `getConfig?ALL` im Quelltext (Grep-Test).
- **SR-12:** Unit-Test: Parser verwirft nicht-whitelisted Keys (inkl. hypothetisch mitgelieferter
  Secret-Keys); kein Log-Aufruf erhält den Response-Body.
- **SR-13:** Unit-Tests: `Access restricted`-Klartext, leeres `{date,time}`-Envelope, `"0"` vs
  `0`, fehlende Keys → jeweils definiertes Ergebnis, keine Exception.
- **SR-14:** Unit-Test: URL enthält nie `user:pass@`; Fallback-Header nur nach restricted-Signal.
- **SR-15:** Unit-Test: `detectFeatures` entfernt bei Config-Widerspruch nur Cover; Force-Override
  hält Capability trotz negativer Config.
- **SR-16:** Unit-Test Throttle-Helfer; Code-Review-Check: kein `setUnavailable` im Config-Pfad.

## 4. Unverändert aus M3 übernommen

SR-01 (keine Creds in URL), SR-02 (keine Creds in Logs), SR-03 (Klartext-HTTP als akzeptiertes
Risiko dokumentiert), SR-08 (Host-Pinning auf den gepairten Host), SR-09 (sanitisierte Fehler).
Der Lesepfad führt keine neuen Schreib-Autoritäten ein; SR-04–SR-07/SR-10 bleiben unberührt.
