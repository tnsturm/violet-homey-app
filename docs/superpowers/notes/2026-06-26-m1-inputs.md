# M1 (LSI) — inputs from M0 live testing (2026-06-26)

Captured while testing M0 on real hardware ("Torstens Homey Pro", SW 13.3.0) against the
real Violet (`http://violet`). M1 is not yet specced/planned; fold these into M1 brainstorming.

## 1. Freshness hardening — solved by `PUMP_LAST_ON` (high priority)

The M0 final review flagged two non-blocking weaknesses in the freshness signal that M1's LSI
will consume:
- pump rising-edge tracked in-memory (`_pumpOnSince`) using `CURRENT_TIME_UNIX` → a backward
  controller-clock step mis-gates freshness;
- `_pumpOnSince` resets on every app restart → warmup is wrongly re-imposed after a reboot/reinstall
  (observed live: reinstalling M0 reset `measurements_fresh` to false for ~one warmup window).

**CONFIRMED in BOTH the real Violet payload and the committed demo fixture** (`test/fixtures/getReadings.all.json`):
the payload exposes `PUMP_LAST_ON` (unix seconds, when the pump last turned ON), and
`CURRENT_TIME_UNIX − PUMP_LAST_ON == PUMP_RUNTIME`:

```
PUMP                = 1
PUMP_LAST_ON        = 1782464400        # unix s
PUMP_LAST_OFF       = 1782417722        # < LAST_ON while running
CURRENT_TIME_UNIX   = 1782471258.752
→ now − PUMP_LAST_ON = 6858 s  ==  PUMP_RUNTIME "01h 54m 18s" (6858 s)   ✓
```

**Recommendation for M1:** refactor `lib/Freshness.js` to derive freshness from the payload, not
in-memory state:

```
fresh = (Number(PUMP) === 1) && ((CURRENT_TIME_UNIX − PUMP_LAST_ON) >= warmupSeconds)
```

clamp the delta at `>= 0`, and remove the `_pumpOnSince` rising-edge tracking in
`drivers/pool/device.js`. One change fixes both findings: a single coherent controller clock
(no skew drift) and a value re-read every poll (survives app restarts/reboots). Update the
`Freshness` unit tests accordingly (feed `pumpLastOn` + `now` instead of `pumpOnSince`).
NOTE: M0 ships the in-memory version; this is an intentional M1 improvement, not an M0 bug.

## 2. LSI inputs

- **Read live from getReadings (confirmed working in M0):** `pH_value`, water temperature via the
  chosen `onewireN` channel (user picks the channel; M0 channel-picker verified).
- **NOT in getReadings → collect as PoolLab/manual device settings in M1:** calcium hardness,
  total alkalinity, cyanuric acid (CYA), TDS/salinity. (Spec §15.)
- LSI must be computed **only while `measurements_fresh === true`**; chemistry warnings
  suppressed/deferred while stale (spec §7). The freshness signal was validated end-to-end on
  real hardware in M0.
- Open question still unresolved (spec §15): stabilized vs unstabilized chlorination (whether to
  apply a CYA correction) — decide during M1 brainstorming.

## 3. Other context

- Default poll interval is now **60 s** (lowered in M0, commit on `origin/main`).
- The Violet payload is field-rich (e.g. `DOS_1_CL_*`, `DOS_4_PHM_*` pH-minus dosing,
  `*_LAST_ON/_LAST_OFF/_RUNTIME` per feature) — relevant to M2 (full read) / M3 (write), not M1.
- M0 is persistently installed on the user's Homey and is intended to replace a manual
  Virtual Device previously fed via Advanced Flows from the Violet (parity is an M2 goal).
